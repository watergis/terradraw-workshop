## Context

The FOSS4G 2026 workshop (branch `docs/foss4g2026`) currently has a single
`docs/workshops/foss4g2026/other-libraries.md` page that shows the
MapLibre→Leaflet migration only as static diff blocks. Terra Draw's biggest
selling point is that the same code works across mapping libraries with only
the adapter changing, so this section will be expanded into one page per
library, each demonstrated with the existing `<terra-draw-editor>` live-editor
widget:

- **Leaflet** — migration page (the one shown live in the workshop): **two
  separate live editors**, one running the MapLibre baseline, one running the
  Leaflet port. Environments are never mixed — each widget instance targets
  exactly one library, which keeps the widget implementation simple.
- **OpenLayers / Mapbox GL JS / Google Maps / ArcGIS SDK** — completed
  examples: Start tab already contains the ported code, Run shows it working
  (no Answer tab).
- API keys (Mapbox, Google Maps, ArcGIS if needed) are baked in at build time:
  **locally from `.env`, on Cloudflare Pages from build environment
  variables**. The materials document how to set the keys up.

**Working branch: `docs/foss4g2026-other-lib`**, created from
`docs/foss4g2026`. Note: `main`'s working tree has an untracked `template/`
that is tracked on `docs/foss4g2026` — verify it matches before checkout
(`git checkout docs/foss4g2026`; if it refuses, diff and remove the untracked
copy first), then branch off.

## 1. Widget changes — `docs/assets/live-editor/live-editor.js` (+ css)

The widget is currently hard-wired to MapLibre (single `IMPORT_MAP`,
`MAPLIBRE_CSS`, `MAP_BOILERPLATE` that creates the map and exposes
`window.map`). Generalize it:

### 1a. Per-library environment registry

Replace the single constants with a `LIBS` registry keyed by library id, each
entry holding: import-map entries, stylesheet URLs, and (optionally) extra
bootstrap. All versions stay pinned in the existing `PINS` object.

Planned CDN sources (each must be verified in the browser during
implementation — CDN/ESM quirks are the known risk area, see
`references/live-editor-widget.md` for past failures):

| lib id | map library | adapter |
|---|---|---|
| `maplibre` (default, unchanged) | esm.sh `maplibre-gl@5.24.0` | unpkg `terra-draw-maplibre-gl-adapter` `.modern.js` |
| `leaflet` | esm.sh `leaflet@1.9.4` (UMD-only → esm.sh), CSS from unpkg | unpkg `terra-draw-leaflet-adapter` `.modern.js` |
| `openlayers` | import-map **prefix** `"ol/": "https://esm.sh/ol@<pin>/"` + `ol.css` | unpkg `terra-draw-openlayers-adapter` `.modern.js` (verify its `ol/*` imports resolve through the prefix) |
| `mapbox` | esm.sh `mapbox-gl@<pin>` + CSS from unpkg | unpkg `terra-draw-mapbox-gl-adapter` `.modern.js` |
| `google` | `@googlemaps/js-api-loader` from esm.sh (loader pattern, no CSS) | unpkg `terra-draw-google-maps-adapter` `.modern.js` |
| `arcgis` | import-map prefix `"@arcgis/core/": "https://js.arcgis.com/<pin>/@arcgis/core/"` (official ESM CDN) + its CSS | unpkg `terra-draw-arcgis-adapter` `.modern.js` |

`terra-draw` itself stays the shared unpkg `terra-draw.modern.js` entry in
every environment so all adapters share one instance.

### 1b. New embed attributes

- `lib="leaflet"` — **single value**, selects one preview environment per
  widget instance. Default: `maplibre` (existing pages untouched). No merging
  of environments.
- `boilerplate="none"` — skips `MAP_BOILERPLATE`; the user code creates the
  map itself into the existing `<div id="map">`. All other-libraries pages use
  this (that's the point — map creation is the part that differs). The
  `#sidebar` div, console hooks, fullscreen etc. stay as is.

### 1c. API-key injection (local `.env` / Cloudflare Pages env vars)

- New gitignored file `docs/assets/live-editor/keys.js` exporting
  `{ mapbox: '...', google: '...', arcgis: '...' }`, generated by a small
  script (`scripts/generate-keys.mjs` or `.py`) that reads **process
  environment variables first** (`MAPBOX_ACCESS_TOKEN`,
  `GOOGLE_MAPS_API_KEY`, `ARCGIS_API_KEY` — this is what Cloudflare Pages
  provides at build time) **and falls back to `.env`** for local development.
  Commit `.env.example`; add `keys.js` and `.env` to `.gitignore`.
- Cloudflare Pages: add the env vars in the Pages project settings and prepend
  the generate-keys step to the build command (before `zensical build`).
  Document the exact build command change in the README.
- The widget lazily imports `keys.js` (tolerate 404 → empty keys) and, before
  running user code, replaces literal placeholders
  `__MAPBOX_ACCESS_TOKEN__` / `__GOOGLE_MAPS_API_KEY__` / `__ARCGIS_API_KEY__`
  in the transpiled source. If a needed key is missing, show a friendly
  console-strip message instead of a raw provider error.
- Keys must be referrer-restricted since the site is public.

## 2. Docs restructure — `docs/workshops/foss4g2026/`

Replace `other-libraries.md` with a directory (`navigation.indexes` is
enabled, so `index.md` can be the section landing page):

```
other-libraries/
  index.md        # current "One API, Many Mapping Libraries" intro + adapter
                  # table + links to each page; sets expectations (only
                  # Leaflet is done live in the workshop, rest are homework)
  leaflet.md      # migration page with TWO editors (see below)
  openlayers.md   # completed example
  mapbox.md       # completed example (key note)
  google-maps.md  # completed example (key note)
  arcgis.md       # completed example (key note)
```

Code files follow the existing convention:

```
code/other-libraries/
  maplibre/start.ts               # the MapLibre baseline (editor 1 on leaflet.md)
  leaflet/{start.ts,answer.ts}    # start = Leaflet skeleton with TODOs, answer = full port
  openlayers/start.ts             # ported code only — no answer.ts → Answer tab hidden
  mapbox/start.ts
  google-maps/start.ts
  arcgis/start.ts
```

**Base example code:** derive a *simplified* version of Exercise 6's answer
(point/linestring/polygon + select mode + one or two sidebar buttons via the
existing `addButton()` pattern, plus the default Atomic Bomb Dome feature) —
small enough that the migration diff stays readable. Unlike the numbered
exercises, these files create the map themselves (no `declare const map`), so
the "what changes / what doesn't" is visible in the code.

**leaflet.md structure (the workshop live demo):**

1. Editor 1: `lib="maplibre"`, `start=../../code/other-libraries/maplibre/start.ts`,
   no answer — the runnable baseline everyone just built.
2. "What changes" section with static diff blocks (imports, map creation,
   adapter line) + admonitions for the gotchas: Leaflet `[lat,lng]` order, no
   `load` event, `tsconfig` `nodenext` note for local SvelteKit use.
3. Editor 2: `lib="leaflet"`, start = the same example with the three
   change-points replaced by `// TODO:` comments (imports/map creation/adapter
   left for the participant to fill in following the diffs above), answer =
   the complete Leaflet port. An admonition notes the start code won't run
   until the TODOs are completed.

**Other library pages:** short intro + local install command
(`pnpm install -D ...`), "what changes vs MapLibre" static code blocks with
library-specific admonitions (API-key setup for Mapbox/Google/ArcGIS), then a
single completed-example editor (`lib="<lib>"`, start only). Each page's
"What's Next?" links to the next library page; `arcgis.md` links to
`../support.md`.

Path rule: pages now live one level deeper, so embeds use
`../../code/other-libraries/<lib>/start.ts`. Update
`maplibre-gl-terradraw.md`'s What's-Next link to `./other-libraries/index.md`.
